# Review

仕様、設計、テスト、セキュリティ、可読性、保守性のレビュー観点を記述します。

## Priority

1. 仕様逸脱、受け入れ条件未達
2. データ破壊、権限、セキュリティ、互換性リスク
3. テスト不足、重要フローの未確認
4. ドキュメント更新漏れ
5. 命名、構成、可読性、保守性

## Rule

- 指摘は具体的なファイルパスと行番号に紐付ける。
- 好みの指摘より、バグや運用リスクを優先する。
- 修正案は小さく実行可能な形で書く。

## Checklist

- [ ] 対象specの受け入れ条件をすべて満たしている。
- [ ] `Out of Scope` に書かれた内容を実装していない。
- [ ] エラー、空データ、境界値、権限不足の扱いがspecと一致している。
- [ ] API変更がある場合、`api.md` と実装が一致している。
- [ ] UI変更がある場合、画面仕様と状態定義が更新されている。
- [ ] ロジック変更に単体テストがある。
- [ ] API、DB、外部連携の変更に統合テストがある。
- [ ] 重要なユーザーフローにE2Eまたは手動確認手順がある。
- [ ] 実行したテストコマンドと結果がPRに残っている。
- [ ] 設計判断が必要な変更はDecision LogまたはADRに記録されている。

## Review Output

レビュー結果は、以下の順に書きます。

1. Findings: 重要度順の指摘。各指摘にファイルパスと行番号を付ける。
2. Open Questions: 仕様判断や追加確認が必要な点。
3. Test Gaps: 実行できなかった、または不足している確認。
4. Summary: 問題がない場合も、確認した範囲を短く書く。
