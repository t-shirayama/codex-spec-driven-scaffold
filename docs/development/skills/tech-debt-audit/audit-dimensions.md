# Audit Dimensions

技術負債監査では、以下の観点を確認する。

## 1. Architectural Debt

- レイヤー違反
- 循環依存
- 責務が集中した巨大モジュール
- 使われていない抽象化
- 実態と合っていないディレクトリ構成

## 2. Code Structure Debt

- 重複ロジック
- 長すぎる関数
- 責務が曖昧なクラスやモジュール
- 命名の揺れ
- マジックナンバーや暗黙知

## 3. Type and Contract Debt

- 型が弱い境界
- 入出力契約が不明確
- バリデーション不足
- APIやイベントのスキーマ不整合

## 4. Test Debt

- 重要パスにテストがない
- 高変更頻度の箇所にテストがない
- 実装詳細に依存した壊れやすいテスト
- スキップされたテスト
- E2Eでしか検知できないリスクが放置されている

## 5. Dependency and Configuration Debt

- 使われていない依存関係
- 同じ目的の依存が複数ある
- 設定の重複
- 環境差分がドキュメント化されていない
- 秘密情報の扱いが曖昧

## 6. Performance and Resource Debt

- 不要な重い処理
- N+1アクセス
- キャッシュ方針の不整合
- リソース解放漏れ
- ホットパス上の同期処理や過剰な変換

## 7. Error Handling and Observability Debt

- 例外の握りつぶし
- 一貫しないエラー形式
- ログ不足
- 監視できない重要処理
- 障害時に原因追跡が難しい構造

## 8. Security Hygiene Debt

- 入力検証不足
- 権限境界の曖昧さ
- 秘密情報の露出
- 安全でないクエリ構築
- 過度に広いアクセス許可

## 9. Documentation Drift

- READMEと実態のズレ
- 仕様と実装のズレ
- コメントとコードの矛盾
- 運用手順の不足
- 重要判断がADRに残っていない

## 10. AI-Generated Code Debt

- 人間が理解しづらい過剰な抽象化
- プロジェクト規約と異なる実装スタイル
- 似たような実装が複数生成されている
- テストなしで生成された複雑なロジック
- 仕様にない振る舞いが追加されている
