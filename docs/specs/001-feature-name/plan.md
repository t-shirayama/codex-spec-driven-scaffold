# Implementation Plan

## Target Areas

| Area     | Expected Change              |
| -------- | ---------------------------- |
| Frontend | 変更がなければ「なし」と書く |
| Backend  | 変更がなければ「なし」と書く |
| Database | 変更がなければ「なし」と書く |
| API      | 変更がなければ「なし」と書く |
| Docs     | 更新対象を書く               |

## Approach

1. 既存仕様、関連コード、テストを確認する。
2. 受け入れ条件を満たす最小の変更を実装する。
3. 単体テスト、統合テスト、E2Eの必要範囲を追加または更新する。
4. spec、画面仕様、API、運用影響のドキュメントを更新する。
5. 実行可能なテストを実行し、結果を記録する。

## Dependencies

- 依存する既存機能、外部サービス、ライブラリ、環境変数を書く。
- 依存がない場合は「なし」と書く。

## Risks

| Risk                   | Impact           | Mitigation                  |
| ---------------------- | ---------------- | --------------------------- |
| 例: 既存フローへの回帰 | 主要導線が壊れる | 関連E2Eを追加または更新する |

## Rollout

- リリース時に必要な手順を書く。
- 段階リリース、feature flag、移行作業が不要な場合は「通常リリース」と書く。

## Verification

- 実行するコマンドを書く。
- 手動確認が必要な場合は、確認手順と期待結果を書く。
